## Почему исключения важны

Программы не всегда работают по счастливому сценарию. Файл может отсутствовать, локатор может быть неверным, сетевой запрос может упасть, а ввод пользователя может оказаться некорректным. Java использует **исключения**, чтобы сигнализировать о таких проблемах.

Обработка исключений важна, потому что она помогает:

- Отделять основную логику от логики ошибок
- Избегать тихих сбоев
- Давать полезные сообщения для отладки
- Восстанавливаться после ожидаемых проблем, если это возможно

Без обработки исключений программа может завершиться с ошибкой без понятного объяснения, что именно произошло.

## Иерархия исключений

Все проблемы в Java наследуются от `Throwable`.

```text
Throwable
├── Error
└── Exception
    └── RuntimeException
```

### Error

`Error` означает серьёзные проблемы на уровне JVM. Обычно прикладной код не должен пытаться их обрабатывать.

Примеры:

- `OutOfMemoryError`
- `StackOverflowError`

### Exception

`Exception` означает проблемы, которые прикладной код может захотеть обработать.

Примеры:

- `IOException`
- `SQLException`
- `ParseException`

### RuntimeException

`RuntimeException` — это особая группа исключений, чаще всего связанная с ошибками в коде.

Примеры:

- `NullPointerException`
- `IllegalArgumentException`
- `IndexOutOfBoundsException`
- `ArithmeticException`

> [!danger] Ключевое правило
> `Error` обычно не обрабатывают. `Exception` обрабатывают или объявляют. `RuntimeException` часто означает баг в коде или некорректные данные.

## Checked и unchecked исключения

Это одна из самых частых тем на собеседованиях.

### Checked исключения

Checked-исключения проверяются компилятором. Ты обязан либо обработать их через `try-catch`, либо объявить через `throws`.

```java
public String readFile(String path) throws IOException {
    return Files.readString(Path.of(path));
}
```

Примеры:

- `IOException`
- `SQLException`
- `ClassNotFoundException`

### Unchecked исключения

Unchecked-исключения являются потомками `RuntimeException`. Компилятор не заставляет тебя их обрабатывать.

```java
public int divide(int a, int b) {
    return a / b; // может выбросить ArithmeticException, если b == 0
}
```

Примеры:

- `NullPointerException`
- `IllegalStateException`
- `NumberFormatException`

> [!tip] Быстрое сравнение
> | Тип | Проверяется компилятором | Типичная причина |
> |---|---|---|
> | Checked exception | Да | Внешняя проблема: файл, БД, сеть |
> | Unchecked exception | Нет | Ошибка в коде или плохой ввод |

Если совсем просто:

- Checked exception -> "что-то снаружи кода может не сработать"
- Unchecked exception -> "код неправильно использует данные"

## try catch и finally

Базовый синтаксис обработки исключений — `try-catch-finally`.

```java
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero");
} finally {
    System.out.println("This block always runs");
}
```

### `try`

Блок `try` содержит код, который может выбросить исключение.

### `catch`

Блок `catch` обрабатывает конкретное исключение.

```java
try {
    Integer.parseInt("abc");
} catch (NumberFormatException e) {
    System.out.println("Invalid number: " + e.getMessage());
}
```

### `finally`

Блок `finally` почти всегда выполняется независимо от того, было исключение или нет. Обычно он нужен для очистки ресурсов.

```java
FileInputStream input = null;
try {
    input = new FileInputStream("data.txt");
    // work with file
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (input != null) {
        try {
            input.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

### Несколько catch-блоков

```java
try {
    String text = args[0];
    int number = Integer.parseInt(text);
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("No argument passed");
} catch (NumberFormatException e) {
    System.out.println("Argument is not a number");
}
```

> [!warning] Порядок важен
> Сначала ставь более специфичные исключения, потом более общие. Если сначала поймать `Exception`, компилятор не разрешит следующий `catch` для `IOException`, потому что он станет недостижимым.

## throw и throws

Эти два ключевых слова часто путают.

### `throw`

`throw` используется внутри метода, чтобы создать и выбросить один объект исключения.

```java
public void setAge(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("Age cannot be negative");
    }
}
```

### `throws`

`throws` используется в сигнатуре метода, чтобы объявить: метод может передать исключение вызывающему коду.

```java
public String loadConfig(String path) throws IOException {
    return Files.readString(Path.of(path));
}
```

> [!info] Короткое правило
> `throw` -> действие внутри метода. `throws` -> объявление в сигнатуре метода.

## Пользовательские исключения

Иногда стандартные исключения слишком общие. В таком случае можно создать собственный класс исключения.

```java
public class InvalidUserDataException extends RuntimeException {
    public InvalidUserDataException(String message) {
        super(message);
    }
}
```

```java
public void register(User user) {
    if (user.getEmail() == null) {
        throw new InvalidUserDataException("Email is required");
    }
}
```

### Когда использовать пользовательские исключения

- Когда ошибка имеет понятный бизнес-смысл
- Когда нужны более хорошие логи и сообщения
- Когда нужен один явный тип для одной области проблем

В автотестах пользовательские исключения делают падения понятнее.

## Try-With-Resources

Если ты работаешь с файлами, потоками, соединениями с БД или другими ресурсами, которые нужно закрывать, используй **try-with-resources**.

```java
try (BufferedReader reader = Files.newBufferedReader(Path.of("data.txt"))) {
    String line = reader.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}
```

Почему это лучше:

- Ресурс закрывается автоматически
- Код короче
- Меньше риск утечки памяти или ресурсов

Это работает с классами, которые реализуют `AutoCloseable`, например:

- streams
- readers/writers
- JDBC connections, statements, result sets

> [!tip] Best Practice
> Предпочитай try-with-resources ручной очистке через `finally`, когда работаешь с закрываемыми ресурсами.

## Обработка исключений в AQA

В автоматизации исключения происходят постоянно. Цель не в том, чтобы их скрывать. Цель — обрабатывать их правильно и оставлять понятные сообщения о падении.

### Частые примеры

```java
try {
    driver.findElement(By.id("login")).click();
} catch (NoSuchElementException e) {
    throw new AssertionError("Login button was not found", e);
}
```

### Типичные исключения в AQA

- `NoSuchElementException` - элемент не найден
- `TimeoutException` - условие ожидания не выполнилось
- `StaleElementReferenceException` - DOM изменился
- `AssertionError` - ожидаемый результат не совпал с фактическим

### Хорошие практики

- Добавляй понятное сообщение при повторном выбрасывании исключения
- Не лови все исключения подряд без причины
- Не скрывай падения тестов пустыми `catch`-блоками
- Лови исключение только тогда, когда можешь восстановиться или улучшить сообщение

### Плохой пример

```java
try {
    driver.findElement(By.id("login")).click();
} catch (Exception e) {
    // ignore
}
```

Это плохо, потому что тест может продолжить работу в сломанном состоянии и упасть позже уже с непонятной причиной.

> [!caution] Пустой catch опасен
> Пустой `catch` скрывает настоящую проблему. В тестах это создаёт ложные успешные результаты или неясные падения. Если ты ловишь исключение, залогируй его, заверни или урони тест с понятным сообщением.

## Вопросы для собеседования

### Топ-10

**1. Что такое исключение в Java?**  
Исключение — это объект, который описывает проблему во время выполнения программы. Java использует его, чтобы прервать обычный поток выполнения и передать управление коду обработки ошибок.

**2. В чём разница между checked и unchecked исключениями?**  
Checked-исключения проверяются компилятором и должны быть обработаны или объявлены. Unchecked-исключения наследуются от `RuntimeException`, и компилятор не заставляет их обрабатывать.

**3. В чём разница между `throw` и `throws`?**  
`throw` используется внутри метода для выбрасывания объекта исключения. `throws` используется в сигнатуре метода для объявления возможных исключений.

**4. Для чего нужен `finally`?**  
Он нужен для кода очистки, который должен выполниться независимо от того, произошло исключение или нет, например для закрытия ресурса.

**5. Можно ли ловить несколько исключений?**  
Да. Можно использовать несколько `catch`-блоков или multi-catch синтаксис вида `catch (IOException | SQLException e)`.

**6. Какой класс является родителем для всех исключений?**  
Технически верхний класс — `Throwable`. `Exception` является родителем для большинства прикладных исключений.

**7. Что такое try-with-resources?**  
Это синтаксис, который автоматически закрывает ресурсы после использования. Он работает с классами, реализующими `AutoCloseable`.

**8. Нужно ли ловить `Exception` везде?**  
Нет. Слишком широкий `catch` скрывает реальные проблемы и усложняет отладку. По возможности лучше ловить конкретные исключения.

**9. Может ли `finally` не выполниться?**  
В обычных ситуациях он выполняется, но в крайних случаях, например при падении JVM или принудительном завершении процесса, он может не выполниться.

**10. Когда стоит создавать пользовательское исключение?**  
Когда проблема имеет понятный бизнес-смысл и отдельный тип исключения делает код и логи понятнее.

### Хитрые вопросы

**1. Является ли `RuntimeException` checked или unchecked?**  
Unchecked. Компилятор не заставляет его обрабатывать.

**2. Можно ли иметь `try` без `catch`?**  
Да, если есть `finally`. Также можно использовать try-with-resources с `catch` и/или `finally`.

**3. Что будет, если и `try`, и `finally` выбросят исключения?**  
Исключение из `finally` может скрыть исходное исключение. Это ещё одна причина, почему try-with-resources безопаснее ручной очистки.

**4. Почему обычно плохо ловить `Throwable`?**  
Потому что тогда ловятся и `Error`, а серьёзные проблемы JVM обычно не нужно обрабатывать как обычные ошибки бизнеса.

**5. В чём разница между `Exception` и `Error`?**  
`Exception` описывает проблемы, которые прикладной код может обработать. `Error` описывает серьёзные системные проблемы, которые обычно не считаются нормально восстанавливаемыми в прикладном коде.
