3. Реальный основной stack.
4. Что понимаешь в C#/.NET ecosystem.
5. Почему интересна эта роль.

Не перечисляй technologies, для которых не готов объяснить хотя бы definition, mechanism и test example.

Если интервьюер сразу указывает на расхождение Java и C#, обозначь его прямо и покажи переносимые знания: OOP, Selenium, API, SQL, test architecture и CI/CD concepts.

## Краткая карта финансового проекта

**Факт:** система собирает, обрабатывает и визуализирует финансовые данные.

**Реконструкция:**

```text
Sources → Kafka → .NET processing
                   ├→ PostgreSQL
                   ├→ MongoDB
                   └→ Redis → API → UI
```

Ключевой E2E answer:

> Я бы использовал уникальный transaction/correlation ID, проверил входной event, запись в PostgreSQL, аналитический document в MongoDB, invalidation Redis, API response и итоговое значение в UI. Для asynchronous processing применил бы polling с timeout. Повторная отправка того же event должна проверить idempotency.

Ключевые risks: duplicate, ordering, rounding, currency, timezone, stale cache, authorization и reconciliation.

## Краткая карта интернет-магазина

**Факт:** система управляет cart, orders, stock, payment и delivery integrations.

**Реконструкция:**

```text
UI → Cart/Order API → PostgreSQL
                    ├→ Payment REST/SOAP
                    ├→ Inventory
                    └→ RabbitMQ → Delivery
Catalog/Pricing ↔ Hazelcast
```

Ключевой E2E answer:

> Я бы проверил total в UI, создание одного order в API и PostgreSQL, stock reservation, payment request с idempotency key, обработку callback, RabbitMQ event, создание shipment и финальный status. Duplicate callback не должен создавать повторную payment или delivery.

Ключевые risks: discount calculation, last item, duplicate order, timeout, SOAP Fault, retry/DLQ, invalidation и cross-browser checkout.

## Быстрые сравнения

| Вопрос | Короткий ответ |
|---|---|
| Kafka vs RabbitMQ | Kafka — distributed log с topics/partitions/offsets; RabbitMQ — broker с exchanges/queues/routing |
| PostgreSQL vs MongoDB | Relational data и constraints против document model и flexible schema |
| Redis vs Hazelcast | Оба могут быть distributed cache; конкретный выбор зависит от ecosystem, data structures и topology |
| REST vs SOAP | Architectural style с HTTP/JSON против protocol с XML envelope и WSDL |
| Unit vs component | Unit изолирует одну небольшую единицу; component проверяет более крупный deployable component через его границу |
| Integration vs E2E | Integration проверяет взаимодействие нескольких частей; E2E проходит полный business flow |
| Mock vs stub | Stub возвращает подготовленные данные; mock дополнительно проверяет interactions |
| Docker vs Kubernetes | Docker создаёт и запускает containers; Kubernetes управляет их deployment и lifecycle в cluster |
| `[Fact]` vs `[Theory]` | Один invariant test против parametrized test с data rows |
| Implicit vs explicit wait | Global polling для element lookup против ожидания конкретного condition |

## Минимум кода

### xUnit и FluentAssertions

```csharp
[Theory]
[InlineData(100, 10, 90)]
public void Discount_ValidData_ReturnsExpected(
    decimal price,
    int percent,
    decimal expected)
{
    var actual = DiscountCalculator.Calculate(price, percent);

    actual.Should().Be(expected);
}
```

### RestSharp

```csharp
var request = new RestRequest("/orders/{id}")
    .AddUrlSegment("id", 123);

var response = await client.ExecuteGetAsync<OrderDto>(request);

response.StatusCode.Should().Be(HttpStatusCode.OK);
response.Data!.Id.Should().Be(123);
```

### Moq

```csharp
var producer = new Mock<IMessageProducer>();
producer
    .Setup(x => x.SendAsync(It.IsAny<OrderCreated>()))
    .Returns(Task.CompletedTask);

await service.CreateAsync(123);

producer.Verify(
    x => x.SendAsync(It.Is<OrderCreated>(m => m.OrderId == 123)),
    Times.Once);
```

Умей объяснить не только syntax, но и границу каждого test.

## Если не знаешь ответ

Используй честный алгоритм:

1. Скажи, какую часть вопроса понимаешь.
2. Обозначь, с чем не работал напрямую.
3. Рассуждай от known principles.
4. Предложи способ проверить предположение.

Пример:

> С конкретной конфигурацией Kafka exactly-once я напрямую не работал. Я понимаю, что обычная at-least-once delivery требует idempotent consumer. Для ответа по конкретному проекту я бы уточнил producer settings, transaction boundaries и способ commit offsets.

Не подменяй неизвестную деталь уверенным предположением.

## Во время интервью

- Сначала ответь прямо, затем добавляй детали.
- Уточняй неоднозначные условия задачи.
- Разделяй fact, assumption и proposal.
- Для scenario questions проговаривай risks, test data, steps и observations.
- Для code questions сначала опиши решение, затем пиши код.
- Если заметил ошибку, спокойно исправь её.
- Не спорь с requirement, пока не уточнил контекст.
- Не используй один и тот же длинный ответ для разных вопросов.

## После интервью

Сразу запиши:

- какие вопросы были заданы;
- где ответ получился слабым;
- какой stack важен работодателю;
- что обещали на следующем этапе;
- какие вопросы нужно разобрать независимо от результата.

Не оценивай результат только по эмоциям. Зафиксируй конкретные evidence: правильные ответы, пропущенные темы и feedback интервьюера.
