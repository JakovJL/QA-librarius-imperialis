## Что известно из CV

**Факт из CV:** проект является e-commerce platform для управления orders, warehouse stock, payment и delivery integrations.

Заявленные QA-задачи:

- проверка REST и SOAP contracts через Swagger и SoapUI;
- проверка payment transactions в PostgreSQL;
- проверка RabbitMQ message distribution;
- проверка Hazelcast caching;
- manual testing cart, discounts и checkout;
- cross-browser и responsive testing;
- анализ и изменение HTTP requests через Postman;
- UI automation на C#, Selenium и xUnit;
- API automation через RestSharp и FluentAssertions;
- запуск UI-tests в Jenkins из Bitbucket repository;
- local environment через Docker Compose;
- unit tests delivery calculation с xUnit и Moq.

## Предметная область

- **Product** — товар с identifier, price и attributes.
- **Cart** — временный набор товаров пользователя.
- **Inventory** — доступное и зарезервированное количество товара.
- **Order** — зафиксированное намерение купить товары.
- **Payment** — попытка списания или подтверждения оплаты.
- **Shipment** — доставка заказа.
- **Discount** — правило изменения цены.
- **Reservation** — временное удержание stock для конкретного order.
- **Idempotency key** — identifier, предотвращающий повторное выполнение одной операции.

Возможные order states:

```text
Created → AwaitingPayment → Paid → Preparing → Shipped → Delivered
   │            │            │
   └────────────┴────────────┴────────► Cancelled
```

Конкретные states должны определяться требованиями. QA проверяет не только каждый status, но и допустимые и запрещённые transitions.

## Техническая реконструкция архитектуры

```text
Browser / UI
     │
     ▼
API Gateway / ASP.NET API
     │
     ├────► Catalog / Pricing ───► Hazelcast
     ├────► Cart service
     ├────► Order service ───────► PostgreSQL
     ├────► Inventory service
     └────► Payment / Delivery adapters
                    │
               REST / SOAP
                    │
          Payment gateway / Logistics

Order service ─────► RabbitMQ ─────► Consumers
```

Вероятная причина использовать RabbitMQ — доставка commands/events между order, payment, inventory и delivery components без жёсткой synchronous связи.

### Жизненный цикл заказа

1. Пользователь добавляет товары в cart.
2. Pricing service рассчитывает price и discount.
3. Checkout проверяет address, delivery method и stock.
4. Order service создаёт order и сохраняет его в PostgreSQL.
5. Inventory резервирует товары.
6. Payment adapter вызывает внешний REST или SOAP service.
7. Payment provider возвращает result сразу или callback позднее.
8. Order status обновляется.
9. RabbitMQ event сообщает другим components об изменении.
10. Delivery service создаёт shipment.
11. UI показывает актуальный status.

### Роль компонентов

| Компонент | Вероятная задача | Что проверяет QA |
|---|---|---|
| PostgreSQL | Orders, payments, reservations | Transactions, constraints, statuses, exact totals |
| RabbitMQ | Commands/events и routing | Exchange, binding, queue, ack, retry, DLQ |
| Hazelcast | Distributed cache | TTL, invalidation, consistency между nodes |
| REST API | Внутренние и внешние operations | Contract, auth, validation, idempotency |
| SOAP | Legacy/external integrations | WSDL, XSD, XML namespaces, SOAP Fault |
| Selenium | User critical path | Cart, checkout, status, cross-browser |
| Docker Compose | Local dependencies | Startup order, network, ports, test isolation |
| Jenkins | Automated pipeline | Trigger, parameters, reports, failure handling |

## REST и SOAP integrations

### REST

Для REST contract обычно используется OpenAPI/Swagger.

Проверки:

- HTTP method и endpoint;
- path/query parameters;
- request/response JSON schema;
- status codes;
- authorization;
- idempotency;
- error body;
- backward compatibility.

### SOAP

Для SOAP contract используется WSDL, а структура XML описывается XSD.

Проверки:

- operation существует в WSDL;
- XML соответствует XSD;
- namespace указан правильно;
- обязательные elements присутствуют;
- SOAP headers и authentication корректны;
- business error возвращается как ожидаемый SOAP Fault;
- timeout и unavailable provider обрабатываются без потери order.

> [!tip] REST vs SOAP
> | Критерий | REST | SOAP |
> |---|---|---|
> | Модель | Architectural style | Protocol |
> | Частый формат | JSON | XML envelope |
> | Contract | OpenAPI | WSDL/XSD |
> | Ошибка | HTTP status и body | SOAP Fault внутри envelope |
> | Инструменты из CV | Swagger, Postman, RestSharp | SoapUI |

→ см. [[09 Основы тестирования API]].

## Проверка RabbitMQ

Ключевые элементы:

- **Producer** публикует message.
- **Exchange** принимает message и применяет routing rules.
- **Binding** связывает exchange с queue.
- **Queue** хранит message до обработки.
- **Consumer** обрабатывает message.
- **Ack** подтверждает успешную обработку.
- **Nack/reject** сообщает о неуспешной обработке.
- **DLQ** хранит messages, которые не удалось обработать обычным путём.

Типичные проверки:

- message попал в нужный exchange;
- routing key направил его в правильную queue;
- payload содержит order ID и корректный event type;
- consumer подтверждает message только после успешной обработки;
- temporary error вызывает ограниченный retry;
- poison message перемещается в DLQ;
- duplicate message не создаёт вторую payment или shipment;
- повторный запуск consumer не теряет незавершённые messages.

> [!tip] RabbitMQ vs Kafka
> | Критерий | RabbitMQ | Kafka |
> |---|---|---|
> | Основная модель | Broker с queues и routing | Distributed event log |
> | Потребление | Message обычно подтверждается consumer | Consumer читает по offset |
> | Routing | Exchanges, bindings, routing keys | Topics, partitions, keys |
> | Частый сценарий | Commands, task queues, гибкая маршрутизация | Event streaming, replay, большой поток данных |
> | Порядок | Зависит от queue/consumer topology | Гарантируется внутри partition |

## Проверка Hazelcast

Hazelcast часто используется как distributed in-memory data grid или cache. В e-commerce там могут находиться catalog data, prices, sessions или reference data, но конкретное назначение в CV не указано.

Проверки:

- cache miss загружает данные из source;
- cache hit возвращает правильное значение;
- TTL очищает устаревшую запись;
- изменение price инвалидирует старый value;
- разные application nodes видят согласованное значение согласно настройкам;
- restart одного node не нарушает ожидаемое поведение;
- недоступный cluster обрабатывается согласно fallback requirements.

Нельзя автоматически считать Hazelcast source of truth. Обычно постоянные business data находятся в database, а cache ускоряет чтение.

## UI и checkout testing

Критический UI flow:

1. Открыть product page.
2. Добавить product в cart.
3. Изменить quantity.
4. Применить discount.
5. Выбрать delivery method.
6. Заполнить customer data.
7. Подтвердить order.
8. Завершить payment flow.
9. Проверить confirmation и order status.

UI checks:

- total пересчитывается после каждого изменения;
- кнопка недоступна при invalid form;
- double click не создаёт два orders;
- browser refresh не дублирует payment;
- ошибки backend показаны понятно;
- данные сохраняются или очищаются согласно требованиям;
- layout работает на поддерживаемых widths;
- виджеты delivery не перекрывают controls;
- critical flow работает в заявленных browsers.

Cross-browser matrix должна основываться на требованиях и статистике пользователей, а не на желании проверить все возможные browsers.

## Automation и local environment

Вероятная local environment через Docker Compose:

```text
tests
├── application/API
├── PostgreSQL
├── RabbitMQ
├── Hazelcast
└── stubs внешних payment/delivery systems
```

Для воспроизводимости нужны:

- фиксированные image versions;
- health checks;
- отдельная network;
- test configuration;
- migrations и seed data;
- cleanup после run;
- stubs для недоступных внешних systems.

Jenkins flow может выглядеть так:

```text
Bitbucket change
      ↓
Build tests
      ↓
Start/choose environment
      ↓
API smoke
      ↓
UI critical path
      ↓
Publish report and screenshots
```

Это модель. CV не сообщает trigger rules, agents, reports и точные stages.

## Критические тестовые сценарии

### Cart и pricing

- минимальное и максимальное quantity;
- price изменился после добавления в cart;
- два несовместимых discounts;
- expired promo code;
- rounding tax и delivery cost;
- product закончился до checkout;
- cart открыт в двух browser tabs.

### Payment

- успешная payment;
- decline;
- timeout;
- provider unavailable;
- duplicate request;
- duplicate callback;
- callback пришёл раньше ответа UI;
- сумма или currency callback не совпадает;
- payment успешна, но order update временно упал.

### Inventory и delivery

- stock reservation успешна;
- reservation expired;
- параллельная покупка последнего item;
- delivery address не обслуживается;
- logistics API возвращает invalid response;
- delivery price изменился;
- отмена должна освободить stock.

## Полный E2E-сценарий

**Сценарий:** пользователь успешно оформляет один товар с online payment.

**Given:** товар доступен, price и delivery rule известны, user авторизован.

**When:** пользователь добавляет товар, выбирает delivery, отправляет checkout и завершает payment.

**Then:**

1. UI показывает правильный total.
2. Order API создаёт один order.
3. PostgreSQL содержит order, items, amount и initial status.
4. Inventory содержит reservation.
5. Payment request имеет правильные amount, currency и idempotency key.
6. Payment callback меняет status один раз.
7. RabbitMQ публикует ожидаемый event.
8. Delivery consumer создаёт shipment.
9. UI показывает confirmation и актуальный status.
10. Повторный callback не создаёт повторную payment или shipment.

## Диагностика падения

Если UI сообщает успех, но order отсутствует:

1. Сохранить browser network request и response.
2. Проверить, какой order ID вернул API.
3. Найти order и payment records в PostgreSQL.
4. Проверить application logs по correlation/order ID.
5. Проверить RabbitMQ queue, unacked messages и DLQ.
6. Проверить response внешнего payment service.
7. Определить первый component с расхождением.

Если UI total неверный:

1. Сравнить values на product page и в cart.
2. Проверить pricing/discount API.
3. Проверить cache entry в Hazelcast.
4. Проверить source data в PostgreSQL.
5. Проверить rounding и порядок применения rules.

## Рискованные формулировки CV

### Postman и browser requests

Postman обычно не анализирует browser traffic автоматически. Корректные варианты:

- найти request в browser DevTools и скопировать как cURL;
- использовать Postman Interceptor;
- вручную воспроизвести request по данным Network tab.

Не следует называть Postman полноценным transparent proxy без уточнения setup.

### Unit tests логики доставки

Если AQA добавляет unit tests в production repository, нужно понимать границу ответственности и code review. Moq изолирует external delivery dependency, но такой test проверяет application logic, а не реальную logistics integration.

### Проверка RabbitMQ

Просмотр message в management UI полезен для диагностики, но не является полной проверкой. Нужны assertions для payload, routing, side effects, retry и idempotency.
