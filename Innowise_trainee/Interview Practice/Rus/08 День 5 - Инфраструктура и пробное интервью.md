## Docker и Docker Compose

**Docker image** — immutable template с application и dependencies.

**Container** — запущенный instance image с собственным process и изолированным environment.

**Docker Compose** — декларативное описание нескольких containers, networks и volumes.

Для local AQA environment полезны:

- application/API;
- PostgreSQL;
- MongoDB или cache;
- Kafka/RabbitMQ;
- stubs внешних integrations;
- test runner.

Пример логики запуска:

```text
docker compose up
      ↓
health checks готовы
      ↓
migrations и seed data
      ↓
API tests
      ↓
UI tests
      ↓
reports и cleanup
```

QA должен понимать:

- image tag фиксирует версию;
- container port отличается от host port;
- services общаются по имени внутри Docker network;
- volume сохраняет данные вне lifecycle container;
- environment variables передают конфигурацию;
- health check показывает готовность, но не полную correctness;
- `depends_on` сам по себе не всегда означает готовность application принять request.

> [!caution] Изоляция tests
> Общая database для параллельных tests создаёт collisions. Используй уникальные identifiers, отдельные schemas/databases или управляемый cleanup.

→ см. [[19 Виртуализация и контейнеризация]] и [[22 Docker и тестовые окружения]].

## Kubernetes

Ключевые объекты:

- **Pod** — минимальная единица запуска одного или нескольких containers.
- **Deployment** — управляет desired number и rollout Pods.
- **Service** — даёт stable network access к Pods.
- **ConfigMap** — обычная конфигурация.
- **Secret** — sensitive configuration; сам объект не заменяет полный secret-management process.
- **Namespace** — логическое разделение resources.
- **Readiness probe** — может ли Pod принимать traffic.
- **Liveness probe** — нужно ли перезапустить зависший container.

Что важно AQA:

- какой environment является target;
- как получить service URL;
- где находятся logs;
- какая application version deployed;
- завершён ли rollout;
- готовы ли dependencies;
- какие ConfigMaps/Secrets выбраны;
- не тестирует ли suite environment во время deployment.

Если UI-tests падают сразу после deployment, сначала проверь rollout и readiness, затем network/DNS и только потом test code.

## CI-CD pipeline

Рекомендуемый порядок stages:

```text
Checkout
   ↓
Restore dependencies
   ↓
Build
   ↓
Static checks / unit tests
   ↓
Deploy or select environment
   ↓
API smoke
   ↓
Integration/component tests
   ↓
UI critical path
   ↓
Publish reports, logs, screenshots
```

Быстрые и дешёвые checks запускаются раньше. Длинные UI regression tests не должны скрывать отсутствие basic API availability.

### GitLab CI-CD для финансового проекта

**Факт из CV:** Selenium UI-tests интегрированы в GitLab CI/CD, а application работает в Kubernetes.

Техническая модель:

1. Pipeline получает test code из GitLab repository.
2. Job восстанавливает NuGet packages и собирает test project.
3. Target environment передаётся через variables.
4. Job ожидает готовность application.
5. Сначала выполняется API smoke.
6. Затем запускаются выбранные UI-tests.
7. Test report и screenshots публикуются как artifacts.

Неизвестные из CV детали: runner executor, tags, browser grid, report tool, trigger rules и deployment strategy.

### Jenkins для интернет-магазина

**Факт из CV:** Jenkins собирает tests из Bitbucket и запускает Selenium scenarios.

Техническая модель:

1. Webhook или расписание запускает Jenkins job.
2. Jenkins checkout делает clone нужной branch.
3. Agent собирает .NET test project.
4. Pipeline поднимает или выбирает environment.
5. Tests получают параметры environment и browser.
6. Выполняются suites по tags/categories.
7. Jenkins сохраняет reports, screenshots и console log.

Важно отличать:

- **CI failure** — build/test job завершился ошибкой;
- **product defect** — actual product behavior нарушает requirement;
- **test defect** — ошибка в test code или test data;
- **environment issue** — system under test или dependency недоступна.

## Конфигурация и безопасность

Хранить в configuration:

- base URLs;
- browser;
- timeouts;
- feature flags;
- database host/name;
- suite selection.

Хранить как protected secrets:

- passwords;
- API tokens;
- client secrets;
- private keys.

Запрещено:

- commit credentials в repository;
- печатать tokens в logs;
- помещать production personal data в test fixtures;
- сохранять полный sensitive response в общедоступный artifact.

Tests должны получать secrets из CI variables или secret manager и маскировать sensitive values.

## Диагностика неуспешного pipeline

Алгоритм:

1. Определить первый failed stage.
2. Проверить, прошли ли build и unit tests.
3. Проверить application version и deployment status.
4. Проверить health/readiness endpoints.
5. Проверить network, DNS и certificate.
6. Проверить test configuration и secrets.
7. Проверить test data и cleanup предыдущего run.
8. Открыть точный failure message, screenshot и request/response.
9. Повторить только failed test в том же environment.
10. Классифицировать: product, test, data или environment.

Перезапуск допустим после сбора evidence, но не является способом анализа причины.

## Flaky tests

**Definition** — flaky test может пройти или упасть без изменения test code и проверяемой функциональности.

Частые причины:

- fixed sleeps и неправильные waits;
- shared test data;
- зависимость от порядка tests;
- race condition;
- unstable locator;
- asynchronous processing без polling;
- environment overload;
- внешняя integration без stub;
- неправильный cleanup;
- current date/time или timezone.

Лечение зависит от причины:

- ждать конкретное condition;
- создавать уникальные data;
- изолировать tests;
- убрать зависимость от execution order;
- использовать stable locator;
- контролировать clock;
- стабилизировать external dependency;
- собирать diagnostics.

Автоматический retry может временно уменьшить шум, но не исправляет причину flaky test.

## Система оценки

Оцени каждый ответ от 0 до 3:

| Балл | Критерий |
|---|---|
| 0 | Нет ответа или ответ неверный |
| 1 | Есть определение, но нет механизма или примера |
| 2 | Есть определение, механизм и простой пример |
| 3 | Есть trade-offs, диагностика и чёткая граница знания |

После двух интервью выпиши только вопросы с оценкой 0–1. Вечером повторяй их, а не весь материал заново.
