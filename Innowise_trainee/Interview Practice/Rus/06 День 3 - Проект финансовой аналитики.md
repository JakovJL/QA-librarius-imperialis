## Что известно из CV

**Факт из CV:** проект является аналитической финансовой экосистемой. Он собирает, обрабатывает и визуализирует финансовые данные из разных источников для мониторинга активов и управления капиталом.

Заявленные QA-задачи:

- manual-проверка данных в PostgreSQL и MongoDB;
- UI testing веб-портала по требованиям из Confluence;
- проверка routing и business logic в ASP.NET WebAPI;
- проверка Redis caching;
- E2E testing финансовой отчётности;
- проектирование regression scenarios;
- UI automation на C#, Selenium и xUnit;
- REST API automation через RestSharp и FluentAssertions;
- component tests с Moq для Kafka-related code;
- запуск UI-tests в GitLab CI/CD после deployment в Kubernetes;
- локальная test environment в Docker.

На интервью каждую обязанность нужно раскрывать по четырём вопросам:

1. Какую проблему проверяли?
2. Какие inputs использовали?
3. Где наблюдали результат?
4. Как локализовали расхождение?

## Предметная область

- **Transaction** — финансовая операция: покупка, продажа, перевод, начисление или списание.
- **Asset** — актив, например денежные средства или ценная бумага.
- **Portfolio** — набор активов клиента или организации.
- **Position** — текущее количество и стоимость конкретного актива.
- **Valuation** — расчёт стоимости актива или portfolio на определённый момент.
- **Report** — агрегированное представление данных за период.
- **Reconciliation** — сопоставление данных между источниками для поиска расхождений.
- **Audit trail** — история изменений: кто, когда и что изменил.

Главные риски финансовых данных:

- потеря или duplicate transaction;
- неправильная валюта;
- ошибка округления;
- неправильный знак суммы;
- неверная дата или timezone;
- повторная обработка одного события;
- рассинхронизация разных storages;
- устаревший cache;
- нарушение прав доступа;
- неполный audit trail.

Для денежных значений обычно используют `decimal` с заранее определёнными precision, scale и rounding rules. `double` может вносить ошибки двоичного представления.

## Техническая реконструкция архитектуры

```text
Внешние источники
       │
       ▼
Ingestion / Adapter services
       │
       ▼
Kafka topics
       │
       ▼
Processing / Normalization services
       │
       ├──────────► PostgreSQL
       │             transactions, users, references
       │
       ├──────────► MongoDB
       │             documents, snapshots, analytical results
       │
       ▼
ASP.NET WebAPI ◄────► Redis
       │
       ▼
Веб-портал: таблицы, фильтры, графики, отчёты
```

В Kubernetes каждый service может работать в одном или нескольких Pods. ConfigMaps могут содержать обычную конфигурацию, а Secrets — credentials и tokens. GitLab pipeline собирает tests и запускает их против выбранного environment.

### Поток финансовых данных

Возможный flow одной transaction:

1. Внешний источник отправляет данные.
2. Adapter проверяет формат и публикует event в Kafka.
3. Consumer читает event и нормализует поля.
4. Business service рассчитывает нужные значения.
5. Transaction сохраняется в PostgreSQL.
6. Аналитический document или snapshot сохраняется в MongoDB.
7. Redis entry инвалидируется или обновляется.
8. API возвращает новые данные.
9. UI обновляет таблицу или график.

Поскольку обработка asynchronous, UI может обновиться не мгновенно. Test должен использовать polling с timeout, а не фиксированный `Thread.Sleep`.

### Роль компонентов

| Компонент | Вероятная задача | Что проверяет QA |
|---|---|---|
| Kafka | Передача событий между services | Payload, key, partition, retry, duplicate, ordering |
| PostgreSQL | Relational и transactional data | Constraints, relations, exact values, transactions |
| MongoDB | Documents и analytical snapshots | Structure, version, nested fields, consistency |
| Redis | Быстрый доступ к часто читаемым данным | TTL, invalidation, cache hit/miss, stale data |
| ASP.NET WebAPI | Business operations и выдача данных | Contract, authorization, validation, errors |
| UI | Аналитические таблицы и отчёты | Filters, sorting, charts, totals, roles |
| Docker | Изолированные dependencies | Configuration, network, startup, cleanup |
| Kubernetes | Deployment и orchestration | Readiness, configuration, logs, scaling effects |

## Рабочий процесс QA

Технически правдоподобный процесс без выдумывания team details:

1. Изучить acceptance criteria в Jira и описание расчётов в Confluence.
2. Выделить sources, transformations и expected outputs.
3. Подготовить test data и уникальный business/correlation ID.
4. Проверить API contract и validation.
5. Отправить или получить входные данные.
6. Найти event и service logs по correlation ID.
7. Проверить PostgreSQL и MongoDB.
8. Проверить Redis или обойти cache.
9. Сопоставить API response с database values.
10. Сопоставить UI с API и требованиями.
11. Зарегистрировать defect с evidence в Jira.
12. После стабилизации сценария решить, на каком test level его автоматизировать.

Автоматизировать через UI следует только пользовательскую ценность, которую нельзя дешевле проверить на API/component level.

## Проверка хранилищ и cache

### PostgreSQL

Подходит для данных с relations, constraints и требованиями к transactions.

Типичные проверки:

- запись создана один раз;
- `transaction_id` уникален;
- сумма, валюта и status совпадают с event;
- foreign keys ссылаются на существующие entities;
- rollback не оставил частичные данные;
- audit fields заполнены;
- aggregation за период возвращает правильный total.

```sql
SELECT transaction_id, amount, currency, status, created_at
FROM transactions
WHERE transaction_id = 'txn-20260717-001';
```

### MongoDB

Подходит для documents с гибкой или вложенной структурой, snapshots и некоторых analytical results.

Типичные проверки:

- document найден по business ID;
- обязательные nested fields присутствуют;
- schema version ожидаемая;
- snapshot содержит правильный reporting period;
- повторная обработка не создала лишний document;
- значения согласованы с PostgreSQL.

MongoDB не следует называть «быстрой заменой PostgreSQL». Выбор зависит от data model, access patterns и consistency requirements.

### Redis

Cache уменьшает latency и нагрузку на основной storage.

Сценарии:

1. **Cache miss:** key отсутствует, service читает database и помещает результат в Redis.
2. **Cache hit:** повторный запрос получает корректное значение из Redis.
3. **TTL:** после истечения времени значение загружается заново.
4. **Invalidation:** после изменения transaction старый key удаляется или обновляется.
5. **Redis unavailable:** система обрабатывает ошибку согласно требованиям и не теряет данные.
6. **Stale cache:** API не должен долго возвращать старую финансовую информацию.

> [!tip] PostgreSQL vs MongoDB vs Redis
> | Критерий | PostgreSQL | MongoDB | Redis |
> |---|---|---|---|
> | Основная модель | Relational tables | Documents | In-memory key-value |
> | Типичная роль | Source of truth для structured data | Flexible documents/snapshots | Cache и быстрый временный доступ |
> | Главный риск | Нарушение constraints/transaction | Несогласованная document schema | Устаревшие или потерянные cache entries |

## Проверка Kafka

Ключевые элементы:

- **Topic** — логический поток events.
- **Partition** — часть topic; даёт parallelism и ordering внутри partition.
- **Offset** — позиция event внутри partition.
- **Consumer group** — consumers, разделяющие обработку partitions.
- **Key** — может определять partition и помогать сохранять порядок для одной entity.
- **Acknowledgement/commit** — фиксация обработанной позиции.

Проверки:

- корректный event schema и обязательные fields;
- правильный topic и key;
- один event обрабатывается ожидаемое количество раз;
- duplicate не меняет результат повторно;
- retry выполняется для временной ошибки;
- poison message не блокирует весь поток;
- consumer восстанавливается после restart;
- порядок сохраняется там, где он нужен;
- offset commit происходит в правильный момент;
- lag не превышает допустимое значение.

Delivery часто строится как at-least-once, поэтому consumer обязан быть idempotent. Exactly-once нельзя обещать без знания конкретной конфигурации и границ transaction.

## Критические тестовые сценарии

### Консистентность данных

1. Сумма из source равна сумме в PostgreSQL.
2. Aggregated value в MongoDB соответствует исходным transactions.
3. API возвращает те же currency и precision.
4. UI total совпадает с API.

### Границы и вычисления

- нулевая и отрицательная сумма согласно business rules;
- максимальная сумма;
- разные currencies;
- округление на границе;
- переход через конец дня, месяца и года;
- daylight saving и timezone;
- пустой reporting period;
- большое количество transactions.

### Надёжность

- повторный Kafka event;
- temporary database failure;
- Redis unavailable;
- consumer restart;
- частичная обработка;
- delayed event;
- два updates одной entity почти одновременно.

### Security

- пользователь без нужной role не видит portfolio;
- один клиент не получает данные другого;
- sensitive values не попадают в обычные logs;
- API проверяет token и permissions;
- audit trail сохраняет критические изменения.

## Полный E2E-сценарий

**Сценарий:** новая transaction должна отразиться в отчёте.

**Given:** существует portfolio `P-100`, reporting currency — BYN, исходный balance известен.

**When:** поступает event с уникальным `transactionId`, amount `100.25 BYN` и временем операции.

**Then:**

1. Event соответствует schema и попал в правильный Kafka topic.
2. Consumer обработал event без error.
3. В PostgreSQL появилась одна transaction с точными values.
4. MongoDB snapshot обновился до ожидаемой версии.
5. Старый Redis key инвалидирован.
6. API через polling вернул обновлённый total.
7. UI после refresh показал правильную сумму и transaction.
8. Повторная отправка того же event не изменила total второй раз.

Evidence для defect report:

- request/event payload;
- correlation ID;
- timestamps;
- SQL result;
- MongoDB document fragment;
- API response;
- screenshot UI;
- relevant logs без credentials и personal data.

## Диагностика падения

Если UI показывает неправильную сумму:

1. Проверить network response в browser DevTools.
2. Повторить API request отдельно.
3. Проверить, не возвращён ли старый Redis value.
4. Сравнить PostgreSQL transaction с MongoDB snapshot.
5. Найти logs processing service по correlation ID.
6. Проверить Kafka event и consumer lag.
7. Определить первый layer, где actual result расходится с expected.

Такой подход локализует проблему. Не начинай расследование только с перезапуска tests.
