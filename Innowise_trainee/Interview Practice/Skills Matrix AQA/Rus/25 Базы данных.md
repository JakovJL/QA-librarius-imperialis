## Вопросы и ответы

### Какие типы баз данных должен знать Automation QA?

**Ответ:**

Relational database хранит data в связанных tables; для запросов обычно используется SQL. PostgreSQL, Oracle Database и Microsoft SQL Server — примеры. Такой тип хорошо подходит, когда важны relationships, transactions и согласованные rules.

NoSQL — широкая группа баз данных с разными models: document, key-value, wide-column и graph. Это не один product, и в нём нет единого query language, transaction model или consistency behaviour, как в relational database. Нужно изучать model и guarantees конкретной database проекта.

### Что такое primary key, foreign key и constraints?

**Ответ:**

Primary key однозначно определяет row и не может быть `NULL`. Foreign key ссылается на primary key или unique key другой table и защищает referential integrity. Например, `orders.customer_id` как foreign key должен ссылаться на существующего customer.

Другие распространённые constraints: `NOT NULL`, `UNIQUE`, `CHECK` и default value. Database должна отклонить data, нарушающие constraint. В tests нужно проверять и valid path, и ожидаемую error для invalid data.

### Чем отличаются SELECT, INSERT, UPDATE и DELETE?

**Ответ:**

`SELECT` читает rows. `INSERT` создаёт rows. `UPDATE` изменяет существующие rows. `DELETE` удаляет rows. `UPDATE` и `DELETE` без корректного `WHERE` могут затронуть много rows, поэтому это рискованные commands.

В automated test нужны parameterised queries или prepared statement. Нельзя собирать SQL из непроверенного text. Следует проверять affected-row count и, когда это делает assertion понятнее, повторно читать result.

### Когда использовать DISTINCT и ORDER BY?

**Ответ:**

`DISTINCT` удаляет duplicate rows из query result. `ORDER BY` задаёт порядок результата по одному или нескольким columns. Без `ORDER BY` SQL не обещает стабильный порядок rows, даже если в test environment result выглядит отсортированным.

Явный порядок нужен, когда test сравнивает list или проверяет первый result. Нельзя применять `DISTINCT`, чтобы скрыть ошибочный join, который создаёт duplicates.

### Чем отличаются INNER JOIN и OUTER JOIN?

**Ответ:**

`INNER JOIN` возвращает rows с совпавшими values с обеих сторон. `LEFT OUTER JOIN` возвращает все rows из левой table и совпавшие rows из правой; columns правой стороны для несовпавших rows имеют `NULL`. `RIGHT OUTER JOIN` работает в обратном направлении. `FULL OUTER JOIN` возвращает совпавшие и несовпавшие rows обеих tables, если database его поддерживает.

Join выбирают по вопросу. Например, left join помогает найти customers без orders. Всегда нужно указывать join condition: отсутствующее или неверное condition может создать неправильный Cartesian product.

### Что такое subqueries, UNION, INTERSECT и EXCEPT?

**Ответ:**

Subquery — query внутри другого SQL statement. Она может дать value, list values или временную result table. `UNION` объединяет совместимые result sets и удаляет duplicates; `UNION ALL` сохраняет duplicates. `INTERSECT` возвращает общие rows, а `EXCEPT` — rows из первого result, которых нет во втором.

Для этих operators нужны совместимые columns, а их точная поддержка зависит от database dialect. Нужно читать documentation database проекта.

### Как работают aggregate functions, GROUP BY и HAVING?

**Ответ:**

`COUNT`, `SUM`, `MIN`, `MAX` и `AVG` вычисляют value для множества rows. `GROUP BY` создаёт один aggregate result для каждой group, например каждого customer или дня. `HAVING` фильтрует groups после aggregation, а `WHERE` фильтрует отдельные rows до grouping.

Например, чтобы найти customers с более чем тремя orders, нужно сгруппировать orders по customer и применить `HAVING COUNT(*) > 3`.

### Как automated test должен подключаться к базе данных?

**Ответ:**

В Java test обычно использует JDBC и driver проекта. URL, user, password, schema и SSL settings должны храниться в environment-specific configuration или secret store, а не в source code. Test account должен иметь только необходимые permissions.

Connections, statements и result sets нужно открывать только на нужную область и надёжно закрывать, например через `try-with-resources`. Connection pool нужен, когда проекту требуются повторные connections; нельзя делить одно изменяемое connection между parallel tests.

### Как сохранять test data безопасными и изолированными?

**Ответ:**

Лучше использовать выделенную test database или schema и уникальные test data, например generated customer ID. Нужно создавать только data, необходимые для scenario, и очищать их в порядке, который соблюдает foreign keys.

Transaction делает группу изменений all-or-nothing, поэтому test может выполнить rollback своей прямой database setup. Но system under test может писать через другое connection и transaction; rollback test connection не отменит эти data. В таком случае нужен API или database cleanup, специально предназначенный для test environment. Нельзя запускать destructive cleanup на production.

### Что такое index и когда он помогает?

**Ответ:**

Index — дополнительная data structure, которая помогает database находить rows для `WHERE` или `JOIN` без полного table scan. Он может ускорить read, но database должна обновлять его при изменениях rows, поэтому появляются storage и write cost.

Нельзя считать, что index используется, только потому что он существует. Перед выводом о performance нужно проверить execution plan в нужной database и realistic data volume.

### Что такое views, stored procedures и triggers?

**Ответ:**

View — именованный query, который application может читать как table; materialized view, если database его поддерживает, хранит обновляемый result. Stored procedure — database-side program code, который вызывают явно. Trigger запускается автоматически при заданном database event, например до или после `INSERT`, `UPDATE` или `DELETE`.

При тестировании нужно знать скрытые effects: trigger может изменить data или создать audit row, а procedure — выполнить commit, validation или изменить несколько tables. Проверяются input, result, errors, permissions и side effects.

### Что такое PL/SQL и как его тестировать?

**Ответ:**

PL/SQL — procedural extension языка SQL в Oracle Database. Он добавляет variables, conditions, loops, exception handling, functions, packages и procedures вокруг SQL statements. Microsoft SQL Server использует T-SQL — это другой dialect.

Для автоматизации PL/SQL нужно вызвать procedure или function с известным input и проверить returned values или output parameters, database changes, errors, transaction behaviour и permissions. Нужны безопасная test schema и очистка созданных data.

### Как связаны NoSQL, Big Data и TimescaleDB?

**Ответ:**

Для NoSQL database проверяется contract проекта с её собственными query API, data model, indexes, partitioning или sharding и consistency guarantees. Test для PostgreSQL нельзя без изменений перенести в MongoDB, Redis, Cassandra или graph database.

Big Data systems часто обрабатывают большие или streaming datasets в distributed storage и computation. TimescaleDB — PostgreSQL extension для time-series data, например metrics или sensor events. Она полезна для этой области, но не заменяет все Big Data systems.

### Как тестировать и диагностировать feature с базой данных?

**Ответ:**

Начинать нужно с business action: вызвать UI или API, затем проверить ожидаемое database state узким read-only query. Проверить stored values, relationships, constraints, timestamps и нужный audit или event record. Database query не должна быть единственной assertion, если важен user-visible result.

При падении test нужно сохранить correlation ID, request data, query, result, database error и time window. Проверить transaction commit или rollback, connection settings, permissions, locks или deadlocks, side effects triggers и procedures, а также eventual-consistency delay, прежде чем считать, что ошибочна product logic.
