## Вопросы и ответы

### Что это такое и что нужно знать?

**Короткий ответ:**

WebDriver создаётся с browser options локально или через RemoteWebDriver. Instance, timeouts и cleanup изолируются для parallel tests.

### Какие практические навыки нужны Automation QA?

**Короткий ответ:**

Использовать Factory для выбора browser, контролировать capabilities, engines, Grid или Selenoid и проверять важные flows в headed mode из-за отличий headless.

### Какие ошибки и риски важны?

**Короткий ответ:**

Избегать global Singleton, leaked sessions, shared drivers, скрытых defaults и отсутствующего `quit()`.

## Требования Intern и Junior

### Как настроить и создать локальный WebDriver?

**Ответ:**

Selenium WebDriver — API для автоматизации браузера. В Java тест обычно создаёт driver конкретного браузера: `ChromeDriver`, `FirefoxDriver` или `EdgeDriver`; все они реализуют интерфейс `WebDriver`.

Современный Selenium Manager обычно находит или загружает совместимый driver. Если project управляет drivers самостоятельно, путь к нему настраивается до создания driver. Version driver должна быть совместима с установленным browser.

```java
ChromeOptions options = new ChromeOptions();
WebDriver driver = new ChromeDriver(options);
driver.manage().window().maximize();
// test actions
driver.quit();
```

`quit()` закрывает всю browser session и освобождает process driver. `close()` закрывает только текущее window, поэтому его недостаточно для общего cleanup.

### Какие browser engines, options и правила cleanup важны?

**Ответ:**

Browsers используют разные engines: Chromium используется Chrome и современным Edge, Gecko — Firefox, а WebKit — Safari. Один user flow может вести себя по-разному из-за rendering, permissions, downloads и browser settings.

Browser options настраивают одну локальную session: headless mode, download directory, window size, language, proxy, certificates или arguments. Options должны создаваться из явной test configuration, а не скрываться внутри test. Для cleanup после failure используется `try/finally` или after hook.

## Требования Middle

### Как работают Factory patterns, capabilities и Remote WebDriver?

**Ответ:**

Driver Factory выбирает driver по значению `chrome`, `firefox` или `edge`. Это не позволяет browser-specific creation попасть в tests. Abstract Factory полезен, когда вместе создаётся семейство объектов: driver, options и supporting services для local или remote environment.

Capabilities описывают запрошенную browser session: browser name, version, platform и supported options. В W3C WebDriver browser-specific options передаются внутри option capability. Старые configurations `multiCapabilities` зависят от tool; для current Selenium parallel sessions создаётся отдельный набор capabilities для каждой session.

`RemoteWebDriver` отправляет WebDriver commands на удалённый server, например Selenium Grid. Test передаёт Grid URL и capabilities, а server создаёт подходящий browser node.

```java
WebDriver driver = new RemoteWebDriver(
    URI.create("https://grid.example/wd/hub").toURL(),
    new ChromeOptions());
```

### Как безопасно запускать headless и parallel tests?

**Ответ:**

Headless mode работает без видимого window. Он полезен в CI, но может отличаться от headed mode по viewport size, GPU behaviour, fonts, downloads, timing, permissions или browser version. Нужно задать фиксированный window size и проверить важное падение в headed mode, прежде чем считать его product defect.

При parallel execution каждому test нужны собственные driver, user, files, ports и test data. Нельзя делить один static driver. Singleton обычно небезопасен для parallel tests; driver на thread или Dependency Injection с test scope безопаснее.

## Требования Senior

### Как работают element wrappers, Multiton, Prototype и logging?

**Ответ:**

Element wrapper добавляет безопасное общее поведение вокруг `WebElement`: например, visible click, понятное сообщение об ошибке, screenshot или explicit wait. Он не должен скрывать business actions или добавлять blind retries.

Multiton хранит один контролируемый instance для каждого key, например один driver на thread ID. Ему нужен надёжный cleanup. Prototype создаёт новые objects копированием настроенного prototype; он нужен только если копирование понятнее обычного создания. Эти patterns не обязательны: лучше выбрать самый простой lifecycle с изолированными sessions.

Selenium и driver logs включаются только на уровне, нужном для диагностики. В artifacts теста собираются browser console logs, driver logs, screenshots, page source и сведения о Grid session. Пароли, tokens и private data в logs сохранять нельзя.
