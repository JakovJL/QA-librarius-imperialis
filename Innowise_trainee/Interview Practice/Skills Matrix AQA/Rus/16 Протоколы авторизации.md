## Вопросы и ответы

### Чем отличаются Authentication и Authorization?

**Ответ:**

Authentication отвечает на вопрос «Кто вы?»: например, пользователь подтверждает identity через password, MFA или внешний identity provider. Authorization отвечает на вопрос «Что вам разрешено?» после определения identity: например, читать report, но не удалять его. API должен проверять authorization на server; скрыть кнопку в UI недостаточно.

### Что такое OpenID, OAuth и OAuth 2.0?

**Ответ:**

OpenID — family identity standards. OpenID Connect (OIDC) — современный identity layer поверх OAuth 2.0. OAuth — delegated authorization: он позволяет client получить limited access без получения user password. OAuth 2.0 заменил OAuth 1.0 и имеет другой protocol design; они не совместимы. OAuth 2.0 сам по себе не определяет user login или identity information.

### Какие роли и endpoints есть в OAuth 2.0?

**Ответ:**

Resource Owner может дать access к protected data. Client — application, запрашивающее access. Authorization Server при необходимости аутентифицирует user и выдаёт tokens. Resource Server хранит protected API и принимает valid access token. Authorization Endpoint начинает interactive authorization request; Token Endpoint меняет authorization code, refresh token или client credentials на tokens. Эти servers могут быть отдельными services или частями одного Identity Server.

### Как работает Authorization Code с PKCE?

**Ответ:**

Client создаёт high-entropy `code_verifier` и передаёт derived `code_challenge` в authorization request. После authorization user получает short-lived authorization code через зарегистрированный redirect URI. Client отправляет code и исходный verifier в Token Endpoint. Server выдаёт tokens, только если verifier соответствует challenge. PKCE защищает public client, если authorization code перехвачен; он не заменяет redirect-URI validation или TLS.

### Что такое access tokens, refresh tokens, scopes и roles?

**Ответ:**

Access token передают Resource Server для вызова protected API. Refresh token, если он выдан, передают только Authorization Server для получения нового access token, поэтому его нужно защищать сильнее. Scope — delegated permission, запрошенное client, например `orders.read`. Role — application authorization attribute, например `admin`; OAuth не определяет roles сам. Tests должны проверять rules и для scopes, и для roles, заданные в application.

### Что такое JWT и как его валидировать?

**Ответ:**

JSON Web Token (JWT) — format token, а не синоним OAuth или OIDC. Signed JWT обычно состоит из трёх Base64URL-encoded parts: header, payload с claims и signature. Decoding не равно validation. Resource Server должен проверить signature через trusted key, а также claims: issuer (`iss`), audience (`aud`), expiry (`exp`) и allowed algorithm. JWT может быть access token или ID token; access tokens также бывают opaque strings.

### Что OIDC добавляет к OAuth 2.0?

**Ответ:**

OIDC позволяет client аутентифицировать user и получать authorization. Он добавляет scope `openid`, ID Token с identity claims, standard discovery metadata и UserInfo endpoint. ID Token нужен client, чтобы узнать об authentication; он не заменяет access token при вызове API. В tests нужно проверять, что каждый token используется только intended recipient.

### Как automated test получает и передаёт token?

**Ответ:**

Для machine-to-machine API tests выделенный test client может применить client-credentials grant в Token Endpoint. Для end-user flow test проходит через Authorization Endpoint с test account и PKCE либо использует approved test-only method. Credentials нужно хранить в CI secret store, запрашивать minimum required scopes, не коммитить tokens и скрывать их в reports и logs. Затем test передаёт access token в HTTP Authorization header.

```java
given()
    .header("Authorization", "Bearer " + accessToken)
.when()
    .get("/api/orders")
.then()
    .statusCode(200);
```

### Как тестировать Authentication и Authorization failures?

**Ответ:**

Сначала valid token подтверждает, что endpoint работает, затем в каждом negative test меняется один controlled condition. Проверяются: отсутствие token, malformed token, expired token, bad signature, wrong issuer или audience, missing scope, wrong role и token другого tenant. Нужно проверить expected status и безопасный error response: обычно `401`, когда authentication отсутствует или невалидна, и `403`, когда identity известна, но permission нет. Response и logs не должны раскрывать token, key или лишние security details.

### Что такое SAML, JWK и identity platforms?

**Ответ:**

SAML — отдельный XML-based federation protocol, часто используемый для enterprise browser single sign-on. JSON Web Key (JWK) описывает cryptographic key, обычно public signing key, опубликованный в JWK Set для JWT validation. Keycloak, Auth0, Microsoft Entra ID (бывший Azure AD) и Active Directory — identity products или directories, а не protocols. QA engineer должен изучить endpoints, claims и test setup выбранной platform, а не ожидать одинакового поведения от всех platforms.

### Какие Senior-проверки Authorization нужны?

**Ответ:**

Senior tests охватывают tenant isolation, privilege escalation, combinations roles и scopes, refresh-token rotation и revocation при поддержке, а также token introspection для opaque tokens при поддержке. Mock Authorization Server помогает воспроизводимо проверить failure cases. CI должен безопасно получать secrets и удалять test users или grants. Audit logs и session monitoring должны показывать полезные events, но не credentials и не полный tokens.
