## Вопросы и ответы

### Что такое Version Control System и зачем она нужна?

**Короткий ответ:**

Version Control System сохраняет изменения файлов, поддерживает совместную работу, хранит историю и позволяет сравнивать или восстанавливать версии. Git является распределённой системой: каждый clone содержит локальную историю. SVN и классический TFS обычно используют централизованную модель.

### Как начать работу в Git и опубликовать изменения?

**Короткий ответ:**

Я задаю имя и email через `git config`, клонирую repository, создаю branch, изменяю файлы, проверяю `git status` и `git diff`, добавляю изменения в staging и создаю commit. Затем синхронизируюсь через `git pull` или fetch с rebase и публикую branch командой `git push -u origin <branch>`.

### Как перенести изменения между branches?

**Короткий ответ:**

Можно выполнить merge и сохранить обе истории, сделать rebase своих commits на новую основу или перенести выбранные commits через cherry-pick. Для незавершённой локальной работы подходит `git stash`. Выбор зависит от правил команды и от того, опубликованы ли commits.

### Чем отличаются Reset, Revert и Restore?

**Короткий ответ:**

`git reset` перемещает указатель branch и может убрать изменения из staging или удалить локальную работу. `git revert` создаёт новый commit, отменяющий старый, поэтому безопаснее для общей истории. `git restore` восстанавливает файлы или убирает их из staging. `reset --hard` может уничтожить незакоммиченные изменения, поэтому сначала нужно проверить status и сделать резервную копию.

### Как просматривать историю и работать с Pull Request?

**Короткий ответ:**

Я использую `git log`, `git show`, `git diff` и при необходимости `git blame`. В Pull Request я описываю изменение, связываю задачу, проверяю CI, отвечаю на review comments и выполняю merge по стратегии проекта. Я не переписываю общую branch без согласования.

### Что такое .gitignore, submodules и Git hooks?

**Короткий ответ:**

`.gitignore` исключает неотслеживаемые локальные или сгенерированные файлы, но не удаляет уже отслеживаемый файл. Submodule хранит ссылку на другой repository и требует отдельной инициализации и обновления. Git hooks запускают scripts при commit или push; общие обязательные проверки лучше выполнять на сервере или в CI.

**Работа с submodule:**

- Добавить его можно через `git submodule add <repository-url> <path>`.
- Клонировать вместе с основным проектом — через `git clone --recurse-submodules`; позже инициализировать — через `git submodule update --init --recursive`.
- Обновить можно через `git submodule update --remote`, если это разрешает политика проекта.
- Для удаления submodule нужно деинициализировать его, удалить рабочую директорию и запись в `.gitmodules`, затем закоммитить изменённый gitlink. Это действие требует внимательного review, потому что изменяет отслеживаемую dependency.

### Как восстанавливать состояние, делать rebase и работать с другими VCS?

**Перезапись локального состояния удалённой веткой:**

Сначала нужно убедиться, что нет нужной незакоммиченной локальной работы. Затем выполнить `git fetch origin` и `git reset --hard origin/<branch>`. `fetch` обновляет сведения об удалённой ветке, а `reset --hard` приводит текущую branch, index и working tree к этому remote commit. Локальные изменения будут уничтожены.

**Rebase:**

`git rebase <upstream>` повторно применяет commits текущей branch поверх новой базы. При конфликте нужно исправить файлы, выполнить `git add <file>` и продолжить через `git rebase --continue`. `git rebase --abort` возвращает branch к состоянию до rebase. Нельзя выполнять rebase уже опубликованной общей истории без согласования с командой.

**SVN и TFS/Azure DevOps:**

SVN обычно централизован: `svn checkout` или `svn co` создаёт working copy, а `svn update` или `svn up` получает изменения с сервера. В SVN merge переносит изменения между branches, а `svn revert` отменяет незакоммиченные локальные изменения. TFVC в классическом TFS также использует центральный сервер; Azure DevOps может хранить как TFVC-, так и Git-repositories. Конкретный порядок merge и rollback зависит от выбранного типа repository и правил проекта.
