## Вопросы и ответы

### Что такое Page Object и что он содержит?

**Ответ:**

Page Object — class, представляющий одну page или screen application under test. Он хранит page-specific locators и actions в одном месте, поэтому UI change обычно требует одной правки. Его public methods описывают user actions или понятные page services, например `loginAs()` или `getErrorMessage()`. Business assertions принадлежат test; Page Object может только проверять, что ожидаемая page загружена.

```java
public final class LoginPage {
    private final WebDriver driver;
    private final By email = By.id("email");
    private final By password = By.id("password");

    public HomePage loginAs(String user, String secret) {
        driver.findElement(email).sendKeys(user);
        driver.findElement(password).sendKeys(secret);
        driver.findElement(By.cssSelector("button[type='submit']")).click();
        return new HomePage(driver);
    }
}
```

### Чем отличаются Page Object, Element Object и Component Object?

**Ответ:**

Page Object представляет whole screen. Element Object оборачивает small reusable control, например custom dropdown или date picker, и скрывает low-level Selenium interaction. Component Object представляет более крупную repeated UI part, например product card, header или table row. Используется composition: page содержит components, а component может содержать elements. Не нужно открывать public `WebElement` fields или дублировать locators в tests.

### Что такое трёхуровневая архитектура тестов?

**Ответ:**

Для простого UI framework используется `test → step/workflow → page`. Test описывает behaviour и делает assertions. Step или Workflow Layer объединяет несколько business actions, например создание order через UI и API. Page Layer работает с одной screen. Такое разделение делает test scenarios читаемыми, но не превращает каждый click в отдельный business step.

### Когда framework нужны четыре или пять layers?

**Ответ:**

Четыре или пять layers нужны, только если они убирают repeated code или делают boundary понятнее. Например, для большого product с repeated complex controls подходит `test → workflow → page → component → element`. Не нужно создавать layers только ради diagram: one-page application с простыми controls станет сложнее поддерживать, если каждое action проходит через пять classes.

### Что такое MVC и когда оно полезно в test automation?

**Ответ:**

Model-View-Controller разделяет data и business rules (Model), presentation (View) и interaction flow (Controller). Test framework не обязан копировать MVC, но идея полезна: test data или API models не должны смешиваться с browser code, а orchestration не должна находиться внутри locator class. Следует выбирать simplest design с понятными responsibilities.

### Каким должен быть хороший automated test?

**Ответ:**

Хороший test читаем, проверяет одно behaviour, не зависит от test order, deterministic и имеет meaningful assertions. Он явно создаёт required data, использует stable waits вместо sleeps, очищает changed state и сообщает достаточно context для диагностики failure. Название test должно говорить о behaviour и expected result, а не об implementation detail, например locator или method name.

### Где должны находиться setup, teardown и configuration?

**Ответ:**

`setUp` создаёт только resources, нужные перед test, например driver, client или test data. `tearDown` закрывает или удаляет их даже после failure. Configuration — base URL, browser, reference to credentials, timeout, locale и environment — берётся из typed configuration и CI variables, а не hard-coded values. Setup, data creation, reporting и business assertions должны быть отдельными responsibilities.

### Как framework должен разделять UI, API, Integration и E2E tests?

**Ответ:**

UI, API, integration, contract и E2E tests нужно размещать в понятных packages или modules; low-level utilities можно разделять только там, где это полезно. API tests должны вызывать API clients напрямую, а не сначала проходить UI. Integration tests проверяют работу real components вместе; E2E tests покрывают critical user journey через boundaries и должны быть немногочисленными. Такая структура даёт быстрый feedback и помогает найти место failure.

### Как Gherkin и обычные tests могут использовать один framework?

**Ответ:**

Gherkin flow может быть таким: `Feature → Scenario → Step Definition → Step/Workflow Layer → Page или API Layer`. Step Definitions переводят readable Gherkin в calls общего workflow code; в них не должно быть locators или большой business logic. JUnit и Cucumber могут использовать одни clients, Page Objects, test-data builders, configuration и reporting. Test adapters нужно держать раздельно, чтобы Cucumber change не ломал обычные JUnit tests.

### Как Clean, Hexagonal, CQRS и frontend architectures влияют на tests?

**Ответ:**

Clean Architecture и Hexagonal Architecture отделяют business rules от external adapters. Ports and Adapters означает, что tests могут вызывать domain port или заменять external dependency test adapter. CQRS разделяет commands, меняющие state, и queries, читающие state; tests должны проверять каждый contract и resulting state. Для SPA, SSR или BFF application выбирается правильная test boundary: browser tests покрывают видимое user behaviour, а API и contract tests проверяют backend и BFF rules без повторения всего через UI.

### Как framework должен поддерживать microservices и разные platforms?

**Ответ:**

Нужен small core для configuration, reporting и common utilities, а затем отдельные reusable modules для Web, Mobile, API, messaging и service-specific clients. Microservice suite требует stable service contracts, version compatibility checks, controlled test data и понятный ownership. Framework должен поддерживать нужные platform adapters: Selenium для Web, Appium для Mobile, HTTP clients для backend, consumers или producers для Kafka и RabbitMQ — без принуждения одного test type работать через другой adapter.

### Как масштабировать framework для parallel runs и dynamic environments?

**Ответ:**

Parallel tests требуют isolated drivers, accounts, test data, files, ports и report artifacts. Нельзя разделять mutable static state между threads. Environment configuration должна быть isolated для каждого run и безопасно переопределяться в CI. Testcontainers может создавать disposable dependency containers для tests; suite должна ждать readiness и очищать resources. CI pipeline определяет, какая suite запускается при change, и предоставляет logs, screenshots и traces для failed jobs.
