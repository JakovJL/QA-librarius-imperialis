## Вопросы и ответы

### Чем отличаются virtual machines и containers?

**Ответ:**

Virtual machine эмулирует complete computer и запускает guest operating system с собственным kernel. Container — isolated process с нужными files и dependencies, но containers одного host разделяют host kernel. VMs дают более сильную operating-system isolation, но используют больше resources и запускаются медленнее. Containers легче и стартуют быстро, но не заменяют VM, когда нужен другой kernel или более сильная isolation.

### Почему containers полезны для разработки и test automation?

**Ответ:**

Containers упаковывают одинаковые runtime, dependencies и configuration для developer, CI job и test environment. Это уменьшает failures вида «works on my machine» и позволяет suite запустить isolated database, API stub или browser service для одного run. Также упрощается cleanup: test data и dependencies можно удалить вместе с environment. Container сам по себе не делает test deterministic: test data, time, network calls и external services всё равно нужно контролировать.

### Что такое Docker image, container, base image, layer и registry?

**Ответ:**

Image — read-only template для создания containers. Container — running или stopped instance image со своим writable layer. Base image — starting image в Dockerfile. Images состоят из immutable filesystem layers, которые можно переиспользовать, что ускоряет builds и distribution. Registry, например Docker Hub или private registry, хранит и распространяет images. Нужно фиксировать trusted image version или digest, а не полагаться на moving tag вроде `latest`.

### Что такое Docker container lifecycle?

**Ответ:**

Container можно создать через `docker create`, запустить через `docker start`, остановить через `docker stop` и удалить через `docker rm`. `docker run` создаёт и запускает новый container одной command. `docker ps -a`, `docker logs` и `docker inspect` помогают проверить state и диагностировать problems. Stopping container не удаляет его, а removing container не удаляет его image или named volume, если это не запрошено отдельно.

### Как написать простой Dockerfile для test environment?

**Ответ:**

Dockerfile описывает, как build image. Нужно выбрать trusted base image, copy только required files, задать working directory и command, а также запускать process non-root user, когда root не нужен. Environment-specific secrets нельзя хранить Dockerfile; их передают approved secret mechanism во время запуска.

```dockerfile
FROM eclipse-temurin:21-jre
WORKDIR /tests
COPY target/test-suite.jar app.jar
RUN useradd --system tester
USER tester
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Как сделать Docker image небольшим, быстрым и безопаснее?

**Ответ:**

Нужно использовать small maintained base image с required runtime. Следует фиксировать versions, применять `.dockerignore`, чтобы исключить build output и secrets, и располагать Dockerfile instructions так, чтобы stable dependency layers кешировались. Multi-stage builds оставляет compilers и build tools вне final image. Где возможно, process запускается non-root user; images сканируются и base images обновляются. Passwords, tokens и private keys нельзя помещать image layer, потому что deleted file может остаться в image history.

### Как containers разделяют networks, data и test dependencies?

**Ответ:**

Containers одной Docker network могут обращаться друг к другу по service name; port на host открывается, только если он нужен host process. Named volume применяется для data, которые должны пережить replacement container, а bind mount — для deliberate local file share. Docker Compose может описать small multi-container test environment, например application, database и mock service. Каждый parallel run требует unique names, data, ports и cleanup, чтобы одна suite не влияла на другую.

### Как containers могут запускать CI agents и browser tests?

**Ответ:**

Jenkins agent может работать в short-lived container, поэтому каждая job получает known toolchain и удаляется после job. Та же идея применима к agents в Azure DevOps, TeamCity или Bamboo. Selenium Grid и Selenoid могут запускать browser sessions в containers, а test runner подключается к ним удалённо. Job должна собрать reports, screenshots, logs и videos до cleanup, а доступ к Docker socket нужно ограничить: он даёт высокие privileges.

### Что делает Ansible в containerized environment?

**Ответ:**

Ansible использует playbooks для configuration hosts и deployment repeatable infrastructure. Например, он может установить container runtime, настроить Docker host, создать users и networks или применить Kubernetes-related setup. Ansible дополняет Docker и Kubernetes: Docker упаковывает application, а Ansible настраивает machines или services, где application работает. Playbook должен быть idempotent: повторный запуск приводит к той же desired configuration, а не создаёт duplicates.

### Что такое container orchestration и чем управляет Kubernetes?

**Ответ:**

Container orchestration управляет многими containers на разных machines: scheduling, desired state, restart, scaling, service discovery, rollout и configuration. Kubernetes — orchestration system, управляющая workload objects, а не отдельными ad-hoc containers. Он полезен, когда команде нужны эти возможности на cluster scale; он добавляет operational complexity и не нужен для одной local test dependency.

### Что такое Kubernetes manifests, Pods, Deployments, Services, ConfigMaps и Secrets?

**Ответ:**

Manifest — YAML или JSON, объявляющий desired state. Pod — smallest deployable Kubernetes unit, он может содержать один или closely related containers. Deployment управляет stateless set replicated Pods и controlled updates. Service даёт stable network endpoint для selected Pods. ConfigMap хранит non-sensitive configuration. Secret предназначен для sensitive data, но Base64 encoding сам по себе не является encryption; нужно настроить access control и encryption at rest. Ingress определяет HTTP routing rules и требует Ingress controller для работы.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: test-api
spec:
  replicas: 2
  selector:
    matchLabels:
      app: test-api
  template:
    metadata:
      labels:
        app: test-api
    spec:
      containers:
        - name: api
          image: registry.example/test-api:1.4.2
          ports:
            - containerPort: 8080
```

### Как работает Kubernetes architecture?

**Ответ:**

Kubernetes cluster состоит из control plane и worker nodes. Control plane хранит и reconciles desired state; его components включают API server, scheduler, controller manager и `etcd` — consistent key-value store для cluster data. Worker node запускает Pods. `kubelet` — node agent, который обеспечивает запуск assigned Pods, а `kube-proxy` реализует часть Service networking во многих clusters. «Master node» — устаревшее informal name; актуальный term — «control plane».

### Как работают Kubernetes networking, remote Docker и diagnostics?

**Ответ:**

Kubernetes networking обычно даёт каждому Pod IP address. CNI plugin реализует cluster network, Service даёт stable access к Pods, а Ingress маршрутизирует HTTP(S) traffic извне cluster. Для диагностики проверяются Pod status и events, container logs, readiness и liveness probes, resource limits, DNS, Service endpoints и network policies. `DOCKER_HOST` или `docker -H` может направить Docker CLI к remote daemon, но доступ должен быть protected, например TLS: открытый unauthenticated Docker daemon может дать полный control над host.

### Как устроены CI/CD pipelines и чем Kubernetes отличается от Docker Swarm?

**Ответ:**

Container CI/CD pipeline build image, запускает tests и scans, публикует immutable version в registry, разворачивает declared version, ждёт readiness и сохраняет evidence для rollback или диагностики. Kubernetes может создавать dynamic CI agents и test namespaces, но pipeline должна их очищать и не разделять secrets. Docker Swarm проще настроить для small deployments; Kubernetes имеет более крупную ecosystem и больше control для complex multi-team workloads, но требует больше operational skill. Выбор зависит от deployment needs, а не от популярности.
